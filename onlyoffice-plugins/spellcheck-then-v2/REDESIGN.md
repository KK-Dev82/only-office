# SpellChecker ภาษาไทย — เอกสารออกแบบใหม่ (Redesign)

> เป้าหมาย: ตรวจคำผิดให้ **แม่นยำ ไม่เดามั่ว** โดยใช้ dictionary ที่เรามีอยู่
> เป็นฐาน (Base) ผสมกับคำที่ผู้ดูแลเพิ่มเองผ่าน M0106 (WordManagement)
>
> เอกสารนี้เป็น "แบบก่อนลงมือแก้โค้ด" — สรุปปัญหา + สถาปัตยกรรมที่เสนอ +
> สิ่งที่ต้องตัดสินใจ เพื่อให้ตกลงทิศทางกันก่อน

---

## 1. หลักการสำคัญ (อ่านก่อน)

ระบบนี้ใช้ตรวจทาน **รายงานการประชุม** ของสำนักงาน สว.
สิ่งที่แย่ที่สุดสำหรับผู้ใช้คือ **ตีคำที่ถูกว่าผิด** (false positive) เพราะ:

- ทำให้ผู้ตรวจเสียเวลาไล่ปิดทีละคำ
- พอเตือนผิดบ่อย ๆ ผู้ใช้จะเลิกเชื่อระบบ แล้วปิดทิ้ง

**หลักการ:** ความแม่นยำ (precision) สำคัญกว่าการจับให้ครบ (recall)
> ยอม "ปล่อยคำผิดหลุดบ้าง" ดีกว่า "เตือนคำถูกผิด ๆ"
> เปรียบเหมือนยามที่เห่าเฉพาะตอนมีขโมยจริง — ดีกว่ายามที่เห่าทุกคนที่เดินผ่าน

## 2. ข้อมูลที่เรามี (Assets) — ของจริง ตรวจสอบแล้ว

| แหล่งข้อมูล | ที่อยู่ | ปริมาณ/โครงสร้าง | ใช้ทำอะไร |
|---|---|---|---|
| **Base dict ไทย** | `only-office/dict/th_TH/words.json` | 51,682 คำ (รวมคำ 2 ตัวอักษร 450 คำ เช่น ใน ไป มา ขอ) | บัญชีคำที่ถูกต้อง (whitelist) + ใช้ตัดคำ |
| **Base dict อังกฤษ** | `only-office/dict/en_US/words.json` | 48,262 คำ | whitelist อังกฤษ |
| **กฎเสียงพ้อง (REP)** | `only-office/dict/th_TH/th_TH.aff` | 10 กฎ (ทร↔ซ, ส↔ซ, รร↔ะ, ฎ↔ฏ) | สร้างคำแนะนำจากเสียงคล้าย |
| **M0106 Dictionary** | DB ผ่าน `api/word-management/dictionary` | คำที่ผู้ดูแลเพิ่ม + variants | เพิ่ม whitelist (ศัพท์เฉพาะ สว.) |
| **M0106 Variants** | ฝังในแต่ละคำ dictionary | `variantWord` → คำที่ถูก + `score` | mapping "คำพิมพ์ผิด → คำถูก" ที่คนคัดเอง |
| **M0106 แก้ไขคำผิด** | `api/word-management/entries?type=WordCorrection` | `word` (คำค้นหา) → `fullWord` (คำแก้) | find→replace ตรงตัว ความมั่นใจ 100% |

**หมายเหตุสำคัญที่เพิ่งพบ:**
- DB **ไม่ได้** seed คำจาก Base dict — DB มีเฉพาะคำที่ผู้ดูแลเพิ่มเองผ่าน M0106
  → Base dict (words.json 51,682 คำ) คือ "พจนานุกรมหลัก" ตัวจริง
  M0106 เป็น "ส่วนเสริม" สำหรับศัพท์เฉพาะที่ Base ไม่มี
- API `dictionary` default limit = 100, ส่ง `limit=-1` = เอาทั้งหมด (ไม่มีเพดานตายตัว)
  → โค้ดปัจจุบันตั้ง `DICT_LIMIT=10000` เอง ซึ่ง **ไม่จำเป็นและเสี่ยงตัดคำทิ้ง**

## 3. ปัญหาของระบบปัจจุบัน (พิสูจน์ด้วยการรันจริง)

### 3.1 มีพจนานุกรม 2 ชุดที่ไม่ตรงกัน
- **Trie** (ใช้ตัดคำ): สร้างจาก Base โดย **ตัดคำสั้นกว่า 3 ตัวอักษรทิ้ง** (minLength=3) + เพิ่ม M0106 ทุกความยาว
- **validWords set** (ใช้เช็คตอนท้าย): Base เต็ม ไม่ตัดอะไร

ผลคือคำหนึ่งอาจ "ถูกตามชุดเช็ค แต่หายจากชุดตัดคำ" → ตัวตัดคำงงและตัดพลาด

### 3.2 การตัดคำพลาด ทำให้คำผิด "ลามไปกินคำถูก" (ตรงกับอาการข้อ 1 ที่แจ้งมา)

รันจริงด้วย dict จริง ประโยค "การประชุม**วันนี**มีวาระสำคัญเรื่อง**เศรษกิจ**"
(พิมพ์ตก: วันนี้→วันนี, เศรษฐกิจ→เศรษกิจ):

```
ตั้ง minLength=3 (ของจริงตอนนี้):  รายงานผิด = ["นีมี", "เศรษ"]
ตั้ง minLength=2 (ทดลอง):          รายงานผิด = ["นี"]
```

- `"นีมี"` = คำผิด "นี" ลามไปกลืนคำถูก "มี" → กลายเป็นก้อนที่ไม่มีความหมาย
- `"เศรษ"` = เหลือแค่เศษของ "เศรษฐกิจ"

> นี่คือที่มาของ **"ระบบมองเป็นประโยคยาว ๆ"** — ยิ่งในเอกสารจริงที่
> มี M0106 คำขยะปน + HTML ตัดตัวอักษรไทยข้ามแท็ก อาการจะยิ่งหนัก

### 3.3 คำแนะนำมั่ว (ตรงกับอาการข้อ 2: "คมนาคม → มา")

เมื่อตัดคำพลาดได้ "เศษคำ" แล้วเอาเศษนั้นไปหา Levenshtein (วัดระยะตัวอักษร):
- เศษ 2 ตัวอักษร เช่น "คม" ห่างจาก "มา" แค่ 2 ระยะ → ระบบเลยแนะนำ "มา"
- **รากปัญหาอยู่ที่ตัดคำผิด ไม่ใช่ตัวแนะนำคำ** แต่โค้ดปัจจุบันไปทุ่ม
  ความซับซ้อนที่ตัวแนะนำคำ (3 แหล่ง + จัดอันดับ metadata) ซึ่งแก้ไม่ตรงจุด

### 3.4 ซับซ้อนเกินจำเป็น (7 ชั้น)
pretokenize → Trie FMM → mergeOrphanOneChar → filter minLength → WordCorrection
pre-scan → คำแนะนำ 3 แหล่ง (Variants/REP/Levenshtein) → จัดอันดับ metadata

หลายชั้นเป็น band-aid ทับ band-aid และมี logic ที่ตายแล้ว (เช่น Path B/C
fallback ที่แทบไม่ถูกเรียก เพราะ trie พร้อมเสมอ)

## 4. สถาปัตยกรรมที่เสนอ (Precision-first, 4 ชั้น)

```
ข้อความ
  │
  ▼
[ชั้น 0] แก้ไขคำผิดตรงตัว (M0106 WordCorrection)         ← มั่นใจ 100%
  │      find→replace แบบ exact, mark ตำแหน่งที่จับแล้ว
  ▼
[ชั้น 1] ตัดคำด้วย Trie เดียว (Base + M0106 รวมกัน, ไม่ filter)
  │      คำที่อยู่ใน dict = ถูก → ข้าม
  ▼
[ชั้น 2] กรองคำที่ไม่รู้จัก (OOV) + การ์ดกันเตือนมั่ว
  │      • ข้ามตัวเลข/อังกฤษทับศัพท์/เครื่องหมาย
  │      • ถ้า token ยาวผิดปกติ = ตัดคำเพี้ยน → ไม่เตือน (กัน blob)
  ▼
[ชั้น 3] แนะนำคำ "เฉพาะเมื่อมั่นใจ" (ไม่มีก็บอกว่าไม่มี)
         1) Variants (M0106)  2) REP เสียงพ้อง  3) Levenshtein ระยะ ≤1
```

### ชั้น 0 — แก้ไขคำผิดตรงตัว (เก็บของเดิม)
- ดึง `WordCorrection` จาก M0106 (`word` → `fullWord`)
- หา substring ตรงตัวในข้อความ จับแล้ว mark ตำแหน่งกันชั้นถัดไปจับซ้ำ
- **ความแม่นยำ 100%** เพราะคนคัดเอง — ให้เป็นชั้นแรกสุด

### ชั้น 1 — ตัดคำด้วยพจนานุกรมเดียว
- **รวม Base (words.json) + M0106 dictionary เป็น Trie ชุดเดียว** ใช้ทั้งตัดคำและเช็ค
  → เลิกระบบ 2 พจนานุกรมที่ไม่ตรงกัน (แก้ปัญหา 3.1)
- **ไม่ตัดคำ 2 ตัวอักษรทิ้ง** (เลิก minLength=3) → ขอบเขตคำแม่นขึ้น (พิสูจน์แล้วใน 3.2)
  - ถ้ากังวลขยะ 1 ตัวอักษรจาก affix: filter แค่ `minLength=2` (ตัดเฉพาะ 1 ตัว) พอ
- คำที่ตัดได้แล้วอยู่ใน dict → ถือว่าถูก ข้าม

### ชั้น 2 — กรองคำไม่รู้จัก + การ์ดกัน false positive
คำที่ไม่อยู่ใน dict (OOV) **ยังไม่เตือนทันที** ต้องผ่านการ์ดก่อน:
- ข้ามตัวเลข ไทย/อารบิก, เครื่องหมาย, ช่องว่าง (เหมือนเดิม)
- ข้ามคำอังกฤษทับศัพท์ที่ผสมในข้อความไทย (heuristic)
- **กัน blob:** ถ้า OOV token ยาวเกินเกณฑ์ (เช่น > 8–10 ตัวอักษรที่ตัดไม่ออกเลย)
  ให้ถือว่า "ตัดคำเพี้ยน" แล้ว **ไม่เตือน** (หรือ flag แยกเป็น "ตรวจสอบเอง")
  → ฆ่าอาการ "มองเป็นประโยคยาว ๆ" ที่ราก

### ชั้น 3 — แนะนำคำเฉพาะเมื่อมั่นใจ
เรียงตามความมั่นใจ หยุดทันทีที่เจอแหล่งที่เชื่อถือได้:
1. **Variants (M0106)** — `variantWord` ตรงกับคำผิด → ใช้ `correctWord` (คนคัดเอง)
2. **REP เสียงพ้อง (th_TH.aff)** — แทนตามกฎแล้ว **ต้องเจอใน dict จริง** ถึงเสนอ
3. **Levenshtein ระยะ ≤ 1** (เข้มกว่าเดิมที่ ≤2) + ความยาวใกล้กัน — ลดการเดามั่ว
- **ถ้าไม่มีคำแนะนำที่มั่นใจ → แสดง "ไม่พบคำแนะนำ"** ไม่ยัดคำมั่ว ๆ ให้

## 5. สิ่งที่ตัดทิ้ง / ลดทอน

| ของเดิม | ทำไมตัด |
|---|---|
| พจนานุกรม 2 ชุด (trie filtered + validWords full) | รวมเป็นชุดเดียว |
| filter `minLength=3` | สาเหตุคำผิดลามกินคำถูก → ลดเหลือ 2 หรือเลิก |
| `mergeOrphanOneChar` | เป็น band-aid ของ minLength — ไม่จำเป็นถ้าแก้ราก |
| `DICT_LIMIT=10000` | ไม่จำเป็น ใช้ `limit=-1` ดึงครบ |
| จัดอันดับ metadata (confidence/frequency/usage) | premature optimization — เอาออกก่อน |
| Levenshtein ระยะ ≤ 2 | ลดเหลือ ≤ 1 เพื่อกันคำแนะนำมั่ว |
| Path B/C fallback (Intl.Segmenter, split-space) | logic ที่แทบไม่ถูกเรียก — ตัดให้โค้ดสะอาด |

## 6. เรื่องที่ต้องตัดสินใจร่วมกัน (Open Decisions)

1. **minLength ของ Trie:** เลือก `2` (กันขยะ 1 ตัวอักษร แต่เก็บ ใน/ไป/มา)
   หรือ `1` (ไม่ filter เลย แต่ต้องมั่นใจว่า words.json สะอาด)
2. **เกณฑ์ความยาว "blob" ที่จะไม่เตือน:** เริ่มที่กี่ตัวอักษร? (เสนอ 8–10)
   และเมื่อเจอ blob จะ "เงียบ" หรือ "flag แยกหมวดให้ตรวจเอง"?
3. **ยังต้องใช้ Levenshtein ไหม** หรือพึ่ง Variants + REP + WordCorrection พอ?
   (ถ้าตัด Levenshtein ออก = แม่นสุด แต่คำแนะนำจะน้อยลง)
4. **คำอังกฤษทับศัพท์ในข้อความไทย** (เช่น "ดิจิทัลทรานส์ฟอร์เมชัน") —
   จะตรวจไหม หรือข้ามทั้งหมด?

## 7. เกณฑ์ทดสอบ (Acceptance Criteria)

ทดสอบด้วยประโยคจริงจากรายงานประชุม ต้องผ่านทั้งหมด:

- [ ] ประโยคปกติที่ทุกคำถูก → **ไม่เตือนเลย** (false positive = 0)
- [ ] "ประธาน", "คมนาคม", "ใน", "ไป", "ที่" → ไม่ถูกตีว่าผิด
- [ ] คำผิดจริง 1 คำกลางประโยค (เช่น "ขนสง") → จับได้ **เฉพาะคำนั้น** ไม่ลามข้าง
- [ ] "วันนี" (ตก ้) → จับเป็น "นี"/"วันนี" ไม่ใช่ "นีมี"
- [ ] คำผิดที่ไม่มีคำแนะนำมั่นใจ → แสดง "ไม่พบคำแนะนำ" ไม่เดามั่ว
- [ ] ศัพท์เฉพาะที่ผู้ดูแลเพิ่มใน M0106 → ไม่ถูกตีว่าผิด

## 8. แผนลงมือ (เสนอเป็นเฟส)

> **ปรับจากแผนเดิม:** ทดสอบแล้วพบว่า "ลด minLength 3→2" **ไม่ใช่การแก้ฟรี** —
> เป็น tradeoff: ขอบเขตคำแม่นขึ้น แต่ทำให้คำผิดที่ขึ้นต้นด้วยพยางค์ซึ่งบังเอิญ
> เป็นคำ 2 ตัวอักษรใน dict (เช่น "ฟาม" จาก "ฟา"+ม, "เศรษกิจ") **หลุดการตรวจ**
> ตรงกับที่คอมเมนต์เดิมในโค้ดเตือนไว้ → จึง **ถอด minLength ออกจากเฟส 1**
> เก็บไว้แก้ตอนรื้อ tokenizer เต็ม (เฟส 2)

- **เฟส 1 (ทำแล้ว — v0.2.0):** *เฉพาะการลด false positive ที่ไม่ทำให้พลาดคำผิด*
  - ✅ การ์ดกัน blob (`MAX_OOV_TOKEN_LENGTH=16`) — token ยาวผิดปกติไม่ flag → แก้อาการ "มองทั้งประโยคเป็นคำผิด"
  - ✅ ลด Levenshtein 2 → 1 (`FUZZY_MAX_DISTANCE=1`) — เลิกแนะนำคำมั่ว เช่น "คม"→"มา"
  - ⏸️ **ไม่แตะ** minLength (tradeoff), ไม่แตะ DICT_LIMIT (ไม่ใช่ต้นเหตุที่แจ้ง)
- **เฟส 2 (ทำแล้วบางส่วน — v0.3.0):** แก้ tokenizer ที่ราก
  - ✅ เขียน `dpMaxMatch` (DP maximum matching) ใน trie.js แทน FMM — แก้ปัญหา "อนุกร" (คำขยะใน dict) ทำให้ "อนุกรรมาธิการ" เพี้ยน → ตอนนี้ตัดเป็น `อนุ | กรรมาธิการ` ถูกต้อง, เร็ว O(n×ความยาวคำ)
  - ✅ ตัด mergeOrphanOneChar (dead code)
  - ⏳ ยังเหลือ: รวมพจนานุกรมเป็นชุดเดียว, ปรับ minLength (ตอนนี้ยัง 3), ตัด metadata ranking
- **เฟส 3:** จัด suggestion ใหม่ตามลำดับความมั่นใจ + "ไม่มีก็บอกว่าไม่มี"
- **เฟส 4:** เก็บกวาด dead code (Path B/C), เขียน test ตามข้อ 7

> เฟส 1 ทำใน plugin ปัจจุบันได้เลย ไม่กระทบ backend / M0106
```
